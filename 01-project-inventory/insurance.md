# Insurance 프로젝트 인벤토리

원본 프로젝트: **insurance** (보험 GA·설계사 업무/고객 관리 중심의 멀티테넌트 웹+데스크톱+모바일 앱)  
이 문서는 **코드 복사 없이** 기술자산보관소 기준으로 기능 후보·재사용성을 먼저 정리한 것이다.

---

## 1. 프로젝트 개요

### 1-1. 한 줄 정의

GA(총괄) 단위 **멀티테넌트**로 설계사·GA 스태프가 **고객(Consultation)**, **차량/청약(자동차 보험)**, **사내/협력 연락·회사록**, **PDF 문서 자동화**, **동의(Consent)**, **뉴스/소식 허브** 등을 한 플랫폼에서 운영한다.

### 1-2. 실행 환경(플랫폼) — 혼동 금지

| 표면 | 설명 | 비고 |
| --- | --- | --- |
| PC 웹 | Vite + React, 동일 `src` | Railway 배포(웹) |
| PC 설치형 | Electron(데스크톱) | `file://` 기준 **API base 주입** 등 웹과 다른 런타임 전제 |
| 모바일(설계사) | `apps/mobile` 등 앱 + OTA | EAS Update 파이프라인 |
| 모바일(고객) | `apps/customer-mobile` | 고객 앱 흐름 |
| API | Node(Express) + PostgreSQL | `server/` — 단일 `index.js`에 엔드포인트/가드 집중 |

**기술자산 관점:** 동일 “기능”이라도 **웹/데스크톱/앱/외부고객 링크**는 입출력·권한·URL 전제가 다르므로 **한 덩어리 표준으로 뭉개지 않는다.**

### 1-3. 핵심 기술·외부 연동(요약)

- **DB:** PostgreSQL  
- **스토리지:** S3 계열(업로드·프리사인 패턴 코드베이스에 존재)  
- **캐시/큐:** Redis(ioredis)  
- **인증:** JWT 기반 세션(역할/GA 스코프 포함)  
- **알림/문자:** 별도 SMS 게이트웨이(`sms-gateway-ec2`)  
- **배포/브랜치:** `develop` → 검증, `main` → 운영(AGENTS.md 기준)

## 2. 사용자·권한 구조

### 2-1. 역할(Role) — 서버에서 정규화되는 값

`server` 측에서 사용하는 역할 집합(요지):

- `SUPER_ADMIN` — 전역(테넌트 가장) 정책·구독·일부 Super 전용 UI
- `GA_ADMIN`, `GA_STAFF` — GA(테넌트) 내부 운영·스태프
- `USER` — GA 소속 **설계사(일반 사용자)**에 해당하는 기본 주체(명명은 프로젝트 convention)
- `INSURER_MANAGER`, `LOSS_ADJUSTER` — 원수사/손해사정사 **뉴스/채널** 등 별도 포털

**재사용 관점:** “역할 문자열 + GA 스코프”가 **API 가드**와 **프론트 라우트 가드**에 동시에 반영되는 패턴이 자산이다.

### 2-2. 멀티테넌트(GA) 스코프

- 사용자는 `gaId`(또는 동등 필드)로 **소속 GA**에 귀속된다.
- 일부 API는 `tenant_ga_id` 등 쿼리/바디로 범위를 넓히는데, **슈퍼관리자만** 신뢰하는 등 **요청당 유효 GA 해석** 로직이 있다(`server/lib/rbacScope.js` 류).

### 2-3. UI 라우트 가드(요지)

`src` 에 `ProtectedRoute`, `SuperAdminRoute`, `StaffRoute`, `GaCarInsuranceRoute`, `InsurerManagerOnlyRoute` 등 **기능/역할/구독**을 조합한 중첩 라우트가 있다.

### 2-4. 구독(Subscription) 정책

문서/구현 흐름 상:

- 기본 `plan=FREE` + `policy_active=false` 로 배포 직후 **전원 통과(차단 없음)** 후,
- 관리자가 **정책 활성화**를 트리거하면 Trial 타이머·감사로그·플래그가 **원자적 트랜잭션**으로 기록되는 설계(리팩터 플랜 문서에 상세).

## 5. 스타터킷에 반영할 구조(이 프로젝트에서 “빼올” 원칙)

아래는 **insurance에 종속된 구현**이 아니라, `03-starter-kit`에 **추상 규칙**으로 흡수할 포인트다.

1. **플랫폼 분리:** PC 웹 / 모바일 웹 / 앱 / 설치형( Electron ) 은 **빌드·Base URL·배포 파이프**가 다르면 동일 “라우트 트리”로 합치지 말 것.
2. **멀티테넌트:** `tenant` 스코프 해석(요청 파라미터 vs 사용자 소속)을 **한 모듈**로 몰고, “슈퍼만 Q 파라미터 신뢰” 같은 **불변식**을 문서화.
3. **권한:** UI 가드만으로 끝내지 말고 **서버 가드 + 거부 감사**를 세트로 본다.
4. **구독/플랜:** “배포 직후 전원 허용 → 정책 활성화로 전환” 같이 **운영 스위치**를 도메인과 분리.
5. **기능 식별자:** `features/<도메인>/` 슬라이스 + 라우트에 주석으로 **권한/구독 이중차단** 위치를 남기는 습관.

## 7. 우선 표준화할 기능 TOP 5(기술자산 우선순위)

1. **INS-RBAC-01** — RBAC + 멀티테넌트 스코프(서버/클라 **이중** 가드 패턴)  
2. **INS-SUB-01** — 구독 정책 스위치 + Trial + 감사(운영 **스위치**로 리스크 분리)  
3. **INS-PLAT-01** — 웹/데스크톱 **API Base·배포** 분리 원칙(다중 클라이언트 런타임)  
4. **INS-AUD-01** — 권한 거부 + 감사(보안·내부 통제)  
5. **INS-CON-01 또는 INS-PDF-01** — **규제/문서** 축에서 외부 설명이 강한 자산(둘 중 먼저: 동의=컴플라이언스, PDF=서류 자동화)
