# 일반 입력값을 개발 산출물로 바꾸는 규칙

이 문서는 일반인이 작성한 프로젝트 설명을 개발자가 바로 사용할 수 있는 산출물로 변환하기 위한 규칙이다.

## 1. 변환 목표

일반 입력폼의 답변을 아래 결과물로 바꾼다.

- 프로젝트 개요
- 사용자/권한 구조
- 플랫폼 구성
- 화면 목록
- 기능 목록
- 추천 기술 스택
- 추천 폴더 구조
- Figma 프롬프트
- Cursor/Codex 개발 지시문
- 1차 MVP 개발 순서

## 2. 일반 표현을 개발 표현으로 바꾸는 기준

### 사용자가 말한 문제

예시:

- 고객이 서류를 보내면 찾기 어렵다.
- 카카오톡과 엑셀로 관리해서 복잡하다.
- 담당자가 고객별 자료를 구분하기 어렵다.

변환:

- 고객별 파일 제출 기능
- 관리자 접수 관리 화면
- 고객별 파일함
- 검색/필터/상태 관리
- 파일 메타데이터 DB 구조

### 사용자가 말한 사람

예시:

- 사장
- 직원
- 고객
- 거래처
- 외부 전문가

변환:

- 관리자
- 담당자
- 고객 사용자
- 파트너 사용자
- 외부 협업자

### 사용자가 말한 화면

예시:

- 고객 목록이 필요하다.
- 파일 올리는 화면이 필요하다.
- 담당자가 확인하는 화면이 필요하다.

변환:

- CustomerListPage
- CustomerDetailPage
- CustomerFileSubmitPage
- StaffSubmissionListPage
- StaffSubmissionDetailPage

### 사용자가 말한 기능

예시:

- 고객이 파일을 올리게 하고 싶다.
- 담당자가 확인하고 보완 요청을 하고 싶다.
- 서명을 받게 하고 싶다.

변환:

- customer-claim
- storage-r2
- personal-message
- signature
- notification

## 3. 플랫폼 결정 규칙

### PC 웹이 필요한 경우

- 관리자가 많은 데이터를 본다.
- 리스트, 검색, 필터, 상세 확인이 많다.
- 엑셀처럼 여러 정보를 한 번에 봐야 한다.

결과:

- PC 관리자 웹 우선
- 좌측 사이드바 + 리스트/상세 구조 권장

### 모바일 웹이 필요한 경우

- 고객이 간단히 제출하거나 확인한다.
- 외부 사용자가 앱 설치 없이 접속해야 한다.
- 현장에서 빠르게 입력해야 한다.

결과:

- 모바일 고객 웹 우선
- 카드형 UI와 큰 버튼 권장

### 모바일 앱이 필요한 경우

- 푸시 알림이 중요하다.
- 카메라/파일/공유 기능이 중요하다.
- 자주 사용하는 고객용 앱이 필요하다.

결과:

- 모바일 앱은 2차 단계로 권장
- 초기에는 모바일 웹으로 검증 가능

### PC 설치형 프로그램이 필요한 경우

- 데스크톱에서 계속 켜두는 업무 프로그램이 필요하다.
- 자동 업데이트가 필요하다.
- 로컬 파일 접근이나 창 제어가 필요하다.

결과:

- Electron 기반 PC 설치형 앱 고려
- 단, 초기 MVP는 웹으로 먼저 검증 권장

## 4. Figma 산출물 결정 규칙

### 구조가 복잡하면

- 서비스 구조도
- 사용자 흐름도
- 권한별 화면 흐름

### 화면이 중요하면

- PC 관리자 와이어프레임
- 모바일 고객 와이어프레임
- 디자인 시스템 초안

### 투자/협업 설명이 필요하면

- 전체 서비스 구조도
- 핵심 사용자 플로우
- 주요 화면 시안

## 5. MVP 범위 결정 규칙

처음 버전에는 반드시 핵심 흐름만 넣는다.

좋은 MVP 예시:

- 고객 등록
- 고객 파일 제출
- 담당자 확인
- 메시지 보완 요청
- 상태 완료 처리

나중으로 미룰 기능 예시:

- 고급 통계
- 결제
- 다국어
- 자동화 배치
- 세부 관리자 설정
- 복잡한 알림

## 6. 출력 형식

일반 입력을 받은 뒤 결과는 항상 아래 순서로 작성한다.

1. 내가 이해한 프로젝트 요약
2. 추천 플랫폼 구성
3. 사용자/권한 구조
4. 필요한 화면 목록
5. 적용할 기술 자산
6. 추천 아키텍처
7. 추천 폴더 구조
8. Figma로 먼저 만들 산출물
9. Cursor 개발 지시문
10. 1차 MVP 개발 순서

## 7. 핵심 원칙

- 사용자가 개발 용어를 몰라도 입력할 수 있어야 한다.
- 입력값은 개발자가 바로 쓸 수 있는 구조로 변환되어야 한다.
- 불명확한 내용은 과도하게 질문하지 말고 합리적인 기본값을 제안한다.
- PC/모바일/앱/설치형 여부는 초기에 반드시 정한다.
- Figma는 복잡한 흐름을 코드 개발 전에 시각화하는 용도로 활용한다.
