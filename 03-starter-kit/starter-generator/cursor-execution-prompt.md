# Cursor 실행 지시문 템플릿

신규 프로젝트 스타터 결과물을 바탕으로 Cursor/Codex에 전달하는 실행 지시문이다.

## 기본 지시문

```text
새 프로젝트를 시작한다.

이 프로젝트는 tech-asset-library의 starter-generator 결과물을 기준으로 만든다.

프로젝트명:
[프로젝트명]

프로젝트 목적:
[프로젝트 목적]

대상 사용자:
[사용자 유형]

선택 플랫폼:
[PC 웹 / 모바일 웹 / 모바일 앱 / PC 설치형 앱 등]

적용할 기술 자산:
[signature, pdf-coordinate, storage-r2, customer-claim, personal-message 등]

기본 원칙:
1. PC와 모바일 화면은 처음부터 분리한다.
2. 공통 로직, 타입, API 클라이언트, 디자인 토큰만 shared로 분리한다.
3. 플랫폼 전용 코드는 플랫폼별 폴더에 격리한다.
4. 권한 구조는 기능 개발 전에 먼저 정의한다.
5. 디자인 시스템 토큰을 먼저 만들고 화면 개발에 적용한다.
6. Figma 시안이 있는 경우 화면 구조와 이동 흐름은 Figma를 기준으로 하되, 코드 구조는 프로젝트 아키텍처 규칙을 따른다.
7. 기존 프로젝트 코드를 통째로 복사하지 말고 기술자산보관소의 기능 문서를 기준으로 최소 단위로 적용한다.

먼저 수행할 일:
1. 프로젝트 폴더 구조를 생성한다.
2. 선택된 플랫폼별 진입점을 만든다.
3. 권한 구조 초안을 만든다.
4. 디자인 토큰 파일을 만든다.
5. 핵심 라우트 구조를 만든다.
6. 1차 MVP 기능 개발 순서를 문서화한다.

절대 하지 말 것:
- PC/모바일을 하나의 거대한 반응형 컴포넌트로 만들지 말 것
- 권한 체크를 프론트에만 두지 말 것
- 디자인 색상을 임의로 하드코딩하지 말 것
- 앱/설치형 전용 코드를 웹 공통 영역에 섞지 말 것
- Figma 시안의 겉모양만 보고 구조 없이 코드를 만들지 말 것
- 초기부터 모든 기능을 한 번에 만들지 말 것

완료 후 보고:
- 생성된 폴더 구조
- 선택한 기술 스택
- 권한 구조 초안
- 라우트 구조 초안
- 디자인 시스템 파일
- 다음 개발 단계
```

## Figma 연계 지시문

```text
Figma 시안을 기준으로 구현한다.

중요:
- Figma는 화면 구조와 사용자 흐름의 기준이다.
- 코드에서는 프로젝트의 디자인 토큰과 공통 컴포넌트 규칙을 따른다.
- PC와 모바일 화면은 분리한다.
- Figma의 화면 이름을 라우트/페이지 이름으로 변환한다.
- 버튼 클릭 이동 흐름은 라우트 구조와 상태값으로 반영한다.
- Figma에 없는 오류/빈 상태/로딩 상태도 반드시 추가한다.

먼저 분석할 것:
1. 화면 목록
2. 사용자 유형별 접근 화면
3. 버튼 클릭 이동 흐름
4. 공통 컴포넌트 후보
5. PC/모바일 분리 기준
6. 디자인 토큰 후보
```

## 1차 MVP 개발 지시문

```text
1차 MVP만 개발한다.

목표:
핵심 사용자 흐름이 처음부터 끝까지 동작하는 최소 기능을 만든다.

범위:
- 로그인/권한 기본 구조
- 핵심 목록 화면
- 핵심 상세 화면
- 핵심 등록/수정 화면
- 파일 또는 메시지 등 선택된 핵심 기능
- 기본 디자인 시스템 적용

제외:
- 고급 통계
- 복잡한 자동화
- 다국어
- 결제
- 세부 관리자 설정
- 대량 처리
- 고급 알림

완료 기준:
- 사용자가 핵심 업무를 끝까지 수행할 수 있음
- PC와 모바일 구조가 분리되어 있음
- 권한 구조가 무너지지 않음
- 디자인 토큰이 적용되어 있음
- 다음 기능을 붙일 수 있는 구조가 유지됨
```
