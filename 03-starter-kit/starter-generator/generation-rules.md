# 스타터 생성 규칙

## 1. 생성 순서

신규 프로젝트 스타터는 아래 순서로 생성한다.

1. 프로젝트 목적과 사용자 정의
2. 플랫폼 범위 결정
3. 권한 구조 정의
4. 핵심 기능 선택
5. 데이터 구조 초안 작성
6. 디자인/UX 방향 결정
7. Figma 산출물 범위 결정
8. 기술 스택 선택
9. 폴더 구조 생성
10. Cursor/Codex 개발 지시문 생성

## 2. 플랫폼 규칙

### PC 웹이 필요한 경우

- 관리자, 담당자, 운영자 화면 중심으로 설계한다.
- 리스트와 상세를 함께 보는 2단 구조를 우선 고려한다.
- 테이블, 필터, 검색, 일괄 작업을 포함할 수 있다.

### 모바일 웹이 필요한 경우

- 고객, 현장 사용자, 간편 제출 UX 중심으로 설계한다.
- 한 화면 한 목적을 원칙으로 한다.
- PC 테이블을 그대로 재사용하지 않는다.
- 카드형 리스트와 바텀시트를 우선 고려한다.

### 모바일 앱이 필요한 경우

- 푸시 알림, 카메라, 파일 첨부, 네이티브 공유 기능 필요 여부를 확인한다.
- 웹뷰 기반인지 Expo/React Native 기반인지 초기에 결정한다.
- 앱 전용 코드는 웹 공통 코드와 분리한다.

### PC 설치형 앱이 필요한 경우

- Electron 등 설치형 런타임을 별도 플랫폼으로 본다.
- 자동 업데이트, 타이틀바, 창 제어, 로컬 파일 접근 여부를 초기에 결정한다.
- 설치형 전용 코드를 웹 공통 영역에 섞지 않는다.

## 3. 권한 생성 규칙

권한은 다음 3가지를 분리해서 설계한다.

- 메뉴 노출 권한
- 데이터 접근 권한
- 액션 수행 권한

각 기능마다 반드시 아래 질문에 답한다.

- 누가 생성할 수 있는가
- 누가 조회할 수 있는가
- 누가 수정할 수 있는가
- 누가 삭제할 수 있는가
- 누가 승인/검수할 수 있는가
- 데이터 범위는 개인/팀/회사/전체 중 어디까지인가

## 4. 기능 선택 규칙

기술자산보관소의 기능 자산은 다음 기준으로 선택한다.

- MVP에 반드시 필요한 기능인지
- 다른 기능의 선행 조건인지
- 플랫폼 구조에 영향을 주는 기능인지
- 권한 구조에 영향을 주는 기능인지
- 디자인/UX 흐름에 영향을 주는 기능인지

초기에는 아래 기능을 우선 검토한다.

- storage-r2
- personal-message
- notification
- signature
- pdf-coordinate
- customer-claim

## 5. 디자인/Figma 생성 규칙

Figma는 다음 목적 중 하나 이상이 있을 때 사용한다.

- 서비스 구조를 시각적으로 설명해야 할 때
- 사용자 이동 흐름을 검증해야 할 때
- PC와 모바일 화면 구조를 초기에 분리해야 할 때
- 투자자/협력사에게 보여줄 시각 자료가 필요할 때
- Cursor 개발 전에 UI 기준을 고정해야 할 때

Figma 산출물은 최소 아래 순서로 만든다.

1. IA 또는 서비스 구조도
2. 사용자 흐름도
3. 핵심 화면 와이어프레임
4. 디자인 시스템 초안
5. 클릭 흐름 프로토타입

## 6. 기술 스택 선택 규칙

### React + Vite + Express

다음 경우에 적합하다.

- 빠른 업무형 SaaS 개발
- Electron PC 앱 확장 가능성
- 서버와 프론트를 단순 분리하고 싶을 때

### Next.js

다음 경우에 적합하다.

- 웹 중심 서비스
- SEO가 필요한 서비스
- 관리자와 사용자 웹을 함께 구성할 때
- Prisma 기반 DB 구성을 사용할 때

### Spring Boot + Next.js

다음 경우에 적합하다.

- 백엔드 안정성이 중요한 서비스
- 복잡한 권한/도메인 로직이 많은 서비스
- Java 기반 운영을 선호할 때

### Expo/React Native

다음 경우에 적합하다.

- 모바일 앱이 핵심일 때
- 푸시, 카메라, 파일 접근이 필요할 때
- Android/iOS를 동시에 고려할 때

## 7. 금지 규칙

- PC와 모바일 화면을 하나의 거대한 반응형 컴포넌트로 시작하지 않는다.
- 권한 구조를 나중으로 미루지 않는다.
- 디자인 시스템 없이 화면을 계속 만들지 않는다.
- 특정 기능의 임시 DB 구조를 전체 표준으로 삼지 않는다.
- Figma 없이 복잡한 화면 흐름을 바로 코드로 만들지 않는다.
- 기술자산보관소의 기존 기능을 검토하지 않고 새로 만들지 않는다.
