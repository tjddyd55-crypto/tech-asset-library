# 아키텍처 규칙

## 핵심 원칙

- 책임 분리를 최우선으로 한다.
- 하나의 모듈은 하나의 주된 이유로만 변경되게 설계한다.
- 프레임워크/DB/외부 API 의존은 도메인 내부로 침투시키지 않는다.

## 계층 기준

- 도메인 계층: 비즈니스 규칙, 상태 전이, 정책
- 애플리케이션 계층: 유스케이스 조합, 트랜잭션 경계
- 인터페이스 계층: API/이벤트/배치 입출력
- 인프라 계층: DB, 캐시, 메시지 큐, 외부 서비스 어댑터

## 금지 사항

- 컨트롤러에서 비즈니스 규칙 직접 처리 금지
- 엔티티에 프레임워크 객체 전달 금지
- 특정 프로젝트 라우트/권한/저장소 구조를 표준으로 고정 금지

## 확장 규칙

- 새로운 기능은 기존 인터페이스를 존중한 채 추가한다.
- 정책 변경이 잦은 영역만 추상화한다.
- 공통화는 최소 2개 이상 실제 재사용 사례 확인 후 진행한다.
